3.4.3. GithubBackend¶

class issueshark.backends.github.GithubBackend(cfg, issue_system_id, project_id)[source]¶

Backend that collects issue from github

get_issues(search_state='all', start_date=None, sorting='asc', pagecount=1)[source]¶

Gets issues from the github API

Parameters

  • search_state – state to be searched (e.g., all)

  • start_date – date from which issues should be collected

  • sorting – sorting of the issues

  • pagecount – page number

property identifier¶

Identifier of the backend (github)

process()[source]¶

Processes the issues from github

  1. Gets the updated_at value of the last issue that was stored (newest updated issue)

  2. Gets issues since this date

  3. Calls for each issue store_issue()

  4. Raises the page counter

store_issue(raw_issue)[source]¶

Transforms the issue from a github issue to our issue model

  1. Transforms the issue to our model

  2. Processes the comments of the issue. See: _process_comments()

3. Processes the events of the issue. See: _process_events(). During this: set back the issue and store it again.

Parameters

raw_issue – like we got it from github

3.4.4. BugzillaBackend¶

class issueshark.backends.bugzilla.BugzillaBackend(cfg, issue_system_id, project_id)[source]¶

Backend that collects data from a Bugzilla REST API

property identifier¶

Identifier (bugzilla)

process()[source]¶

Gets all the issues and their updates

  1. Gets the last stored issues updated_at field

  2. Gets all issues that was last change since this value

  3. Processes the results in 50-steps

  4. For each issue calls: issueshark.backends.bugzilla.BugzillaBackend._process_issue()
